# Forum v1.13 Context Pressure Design

> Make Forum the local orchestration layer that decides what context every agent sees,
> how much prompt pressure a run is allowed to create, and how that pressure is
> witnessed in the ledger.

| | |
|---|---|
| Project | Forum |
| Release target | v1.13 |
| Status | Design written, pending operator review |
| Location | `C:/dev/public/forum` |
| Primary goal | Token and context management for local, model-agnostic orchestration |
| Related tools | Index context envelopes, Telos action receipts, Forum causal ledger |

## 1. Why this is the next milestone

The larger objective is for Forum, Index, and the Project Telos tools to replace
generic agent orchestration systems with a local, receipt-backed architecture. Forum
already has the ledger, routing, per-task context, delivery checks, model-call budgets,
resume, receipts, HTTP, CLI, and MCP. The missing control point is context pressure:
Forum can cap upstream output and count model calls, but it cannot yet express a full
run-level context budget, prove how much context was admitted, or distinguish retained,
trimmed, and omitted context across request-level planning, per-task dispatch, and data
flow.

That gap matters more than another prose polish pass. A local orchestration system beats
larger frontier calls only when it spends context deliberately: small routed packets,
fresh task-specific context, ledger joins, and explicit omission records instead of
whole-workspace prompt dumps.

## 2. Scope

This release adds a deterministic context-pressure layer. It does not train models,
implement a new memory system, or replace Index. Index remains the peer brain and
context-envelope producer. Forum becomes the consumer and enforcer: it admits, trims,
or omits context under a declared budget and witnesses every decision.

In scope:

- approximate token accounting from byte length using the same model-agnostic rule Index
  documents for context envelopes;
- a new `ContextBudget` config that is separate from `RunBudget`;
- deterministic admission for request-level context, per-task context, and upstream
  data injection;
- witnessed `context_budget` ledger entries when context is retained, trimmed, omitted,
  or passed under pressure;
- summary and bench metrics for context pressure;
- CLI, HTTP, and MCP surfaces for the budget knobs;
- docs and examples that show Index envelope output being consumed by Forum through the
  existing `ContextProvider` seam.

Out of scope for v1.13:

- semantic compression generated by a model;
- automatic calls to `index` as a subprocess;
- persistent vector memory;
- model-specific tokenizers;
- changing the `ContextProvider` protocol to depend on Index package types;
- delivery/persona policy beyond preserving the existing delivery checks.

## 3. Requirements

1. Forum must stay zero-dependency in core.
2. Budgeting must be deterministic and replayable.
3. Context accounting must be model-agnostic and comparable with Index envelopes.
4. A budget decision must never silently drop material. It must either retain, trim, or
   omit with a reason that lands in the ledger.
5. Existing behavior with no `ContextBudget` must remain compatible except for new
   summary fields defaulting to zero.
6. CLI, HTTP, MCP, Python API, README, USAGE, and CHANGELOG surfaces must stay aligned.
7. Tests must cover core budgeting, dispatcher integration, orchestrator integration,
   summary metrics, and surface parsing.

## 4. Architecture

Add a small pure module:

```text
forum.context_budget
```

Primary types:

```text
ContextBudget
  max_total_tokens: int | None
  max_request_tokens: int | None
  max_task_tokens: int | None
  max_upstream_tokens: int | None
  bytes_per_token: int = 4

ContextPressure
  source: "request" | "task" | "upstream"
  label: str
  original_bytes: int
  admitted_bytes: int
  original_tokens: int
  admitted_tokens: int
  action: "retained" | "trimmed" | "omitted"
  reason: str

ContextBudgetMeter
  admitted_tokens_total: int
  pressures: list[ContextPressure]
```

Core functions:

```text
approx_tokens(text, bytes_per_token=4) -> int
apply_context_budget(source, label, text, budget, meter) -> tuple[str, ContextPressure]
```

The module works only on strings and integers. It has no ledger dependency. Dispatcher
and engine code decide when to witness the returned pressure record.

## 5. Data Flow

Request-level context:

1. `Orchestrator.submit` asks the `ContextProvider` for context for the whole request.
2. If a `ContextBudget` exists, Forum applies `max_request_tokens` and
   `max_total_tokens`.
3. Forum witnesses a `context_budget` entry when the returned context is non-empty or
   when a non-empty context is omitted by budget.
4. The admitted context, not the raw context, is fed into the coordinator prompt.
5. The request-level `context` ledger entry stores the admitted context only.

Per-task context:

1. `dispatch_plan` asks the same `ContextProvider` for each task instruction.
2. Forum applies `max_task_tokens` and remaining total budget.
3. The admitted slice is appended to the task prompt and witnessed as the existing
   `context` entry.
4. A `context_budget` entry records retained, trimmed, or omitted state.

Upstream data injection:

1. `augment_with_upstream` keeps the existing character cap behavior by default.
2. When a `ContextBudget` is supplied, upstream injection uses
   `max_upstream_tokens` and remaining total budget.
3. A trimmed upstream injection keeps the full upstream result in the ledger and marks
   the injected prompt slice with a short deterministic truncation marker.
4. A `context_budget` entry records the pressure decision for each upstream result
   that was considered.

## 6. Ledger Contract

`context_budget` entries use a stable payload shape:

```json
{
  "schema": "forum.context-pressure/v1",
  "source": "task",
  "label": "T2",
  "action": "trimmed",
  "reason": "max_task_tokens",
  "original_bytes": 2400,
  "admitted_bytes": 800,
  "original_tokens": 600,
  "admitted_tokens": 200,
  "remaining_total_tokens": 1200
}
```

Reasons are a closed vocabulary:

- `under_budget`
- `max_request_tokens`
- `max_task_tokens`
- `max_upstream_tokens`
- `max_total_tokens`
- `empty`

Forum should not witness raw omitted context. If content is omitted, the ledger records
only byte and token counts plus reason. If content is trimmed and then admitted, only the
admitted slice is stored in the normal `context` or prompt path.

## 7. Surface Changes

Python API:

- `Orchestrator.submit(request, budget=None, context_budget=None)`
- `dispatch_plan(..., context_budget=None)`
- `build_orchestrator(...)` accepts no new required argument.

CLI:

- `forum submit TEXT --context-token-budget N`
- `forum submit TEXT --request-context-token-budget N`
- `forum submit TEXT --task-context-token-budget N`
- `forum submit TEXT --upstream-token-budget N`

HTTP:

`POST /submit` accepts:

```json
{
  "request": "...",
  "context_token_budget": 4000,
  "request_context_token_budget": 1000,
  "task_context_token_budget": 800,
  "upstream_token_budget": 800
}
```

MCP:

- `submit` and `forum.submit` accept the same optional fields.
- Tool descriptions mention budgeted context admission.

Receipts:

- `project-telos.action-receipt/v1` receives a compact `context_budget` block with
  configured limits and observed admitted tokens.

## 8. Reporting

`forum.report.summarize` adds:

- `context_budget_checks`
- `context_budget_trimmed`
- `context_budget_omitted`
- `context_tokens_original`
- `context_tokens_admitted`
- `context_tokens_saved`

`forum bench` automatically compares those numeric fields, so a change can prove that a
run carried less context pressure while preserving verification.

## 9. Error Handling

- Negative budgets raise `ValueError` at construction or parser time.
- `bytes_per_token` must be positive.
- Budgeting never raises because text is too large; it trims or omits.
- A context provider exception keeps the existing behavior: the caller path witnesses
  failure where appropriate and does not turn omitted provider output into invented
  context.
- If total budget is exhausted, later context slices are omitted with
  `reason=max_total_tokens`; the run can still proceed without that context.

## 10. Testing Plan

Core tests:

- approximate-token calculation counts UTF-8 bytes, not code points;
- retained, trimmed, and omitted pressure decisions;
- total budget is consumed across multiple calls;
- invalid budget values raise clear errors.

Dispatcher tests:

- per-task context is trimmed and witnessed;
- omitted per-task context does not appear in the executor prompt;
- upstream data injection is budgeted and keeps full upstream result in the ledger.

Engine tests:

- request-level context is budgeted before planning;
- context-budget entries chain to the request, plan, or task path;
- runs with no `ContextBudget` preserve existing behavior.

Surface tests:

- CLI parses budget flags into `ContextBudget`;
- HTTP `/submit` accepts budget fields;
- MCP `submit` and `forum.submit` route budget fields through the same HTTP surface;
- receipts include context-budget metadata.

Reporting tests:

- summary counts checks, trims, omissions, original/admitted/saved tokens;
- bench compares the new metrics.

Targeted verification:

```bash
python -m pytest tests/test_context.py tests/test_dispatch.py tests/test_budget.py tests/test_report.py tests/test_cli.py tests/test_http_surface.py tests/test_mcp_surface.py
forum status --json
forum doctor --json
```

4. Add summary and bench metrics.
5. Add CLI, HTTP, MCP, and receipt fields.
6. Update README, USAGE, ARCHITECTURE, CHANGELOG, and examples.
7. Run the targeted verification slice.

## 12. How this advances the larger objective

This milestone does not claim that Forum replaces every orchestration tool by itself.
It moves the system toward that state by making Forum the local authority for context
admission and pressure accounting. Once context pressure is measured and witnessed,
the next milestones can build on a stronger base:

- delivery policy can become domain-aware without dropping evidence;
- council and critic workflows can spend context intentionally;
- Index context envelopes can be consumed as compact, budgeted state rather than
  copied wholesale;
- Telos action receipts can show not just what was done, but what context was allowed
  to shape the work.
